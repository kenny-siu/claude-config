# Reword Using Plain English Principles

Reword the text the user provides using the plain English principles below. Keep the original meaning, intent, key facts, names, dates, and technical details accurate.

## Output Format

1. Reworded text
2. Optional short notes on the most important changes

---

## Plain English Principles

### Before You Start

Identify what the reader needs to know or do after reading. Frame it as: "The user need is to [X] so they can [Y]." Ask the user to confirm if unclear.

### Word Choice

- Use simple, common words: "buy" not "purchase", "help" not "assist", "about" not "approximately"
- Prefer short words over long ones
- No jargon or buzzwords — if a technical term is essential, explain it on first use
- Use "and" instead of "&", "to" instead of dashes in ranges (for readability and screen readers)

### Voice and Tone

- **Active voice** over passive: "We will send you a letter" not "A letter will be sent to you"
- **Address the reader as "you"** to keep it direct and conversational
- **Serious but human** — be specific and informative without sounding like a machine
- **Gender-neutral** — use "they", "their", "them"
- No offensive words or terms

### Sentences and Paragraphs

- **Sentences:** Aim for 25 words or fewer
- **Paragraphs:** No more than 5 sentences each
- Write for a **9-year-old reading age** — research shows even specialists prefer plain English because it's faster to process

### Structure

- **Inverted pyramid:** Most important information at the top, lesser details below
- **Front-load:** Put the most important keywords first in titles, headings, and bullet points (readers scan in an F-shape)
- **Heading hierarchy:** Use H2–H3 to break up content; don't skip levels
- **Bulleted lists** to break down information into scannable chunks
- **No FAQs** — write content that meets user needs in context
- **No block capitals** for large amounts of text (harder to read)

### Logical Flow by User Need

Choose a structure that matches what the reader needs:

| User Need | Flow |
|-----------|------|
| Finding an answer | Question > Answer > Understanding > Next steps |
| Reviewing a proposal | Context > Problem > Proposal > How it solves the problem > Restrictions > Future considerations |
| Learning a skill | Goal > Prerequisites > Step-by-step instructions > Troubleshooting > Success criteria |
| Making a choice | Criteria > Options > Comparison > Recommendation |
| Reviewing an event | Summary > Timeline > Root cause > Resolution > Prevention |

### Transformation Examples

| Before (complex) | After (plain) |
|-------------------|---------------|
| "Hazardous waste — new process" | "How to dispose of hazardous waste in your area" |
| "Potato guidance" | "How to grow potatoes" |
| "The recently implemented categorical standardisation procedure on waste oil should not be applied before 1 January 2025." | "Do not use the new waste oil standards before 1 January 2025." |
| "Using and submitting your business expenses." | "Submitting your business expenses." |
| "At the activity centre: you can swim, you can play, and you can run." | "At the activity centre you can: * swim * play * run" |
| "Regarding the server migration, we have noticed lag and suggest a cloud transition next month." | "**Proposal: Moving to the cloud** — Our current servers are slow (the problem). We propose moving to the cloud (the solution) to speed up our work." |
